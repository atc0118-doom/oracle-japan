# ORACLE JAPAN — 国内生活リスク指数(実験ツール)

ORACLE(世界リスクインテリジェンス)の日本国内特化版。JAPAN NOW(中立ニュース集約)とは
役割が異なり、こちらは「日本での暮らしに直結するリスクをスコア化する」のが目的。

## 設計判断

- **配管はORACLE本体から流用**: 10分キャッシュ、Upstash Redisによる24Hデルタ永続化(未設定でも
  無言でno-opする)、`isBaseline`による「ライブでないデータをライブと偽らない」フォールバック設計。
- **災害カテゴリだけ構造化データ駆動**: JMA公式警報(都道府県別・特別警報/警報/注意報の重み付け)
  と地震Atomフィード(震度5弱以上の非定時速報のみ)から算出。ここはキーワード推測ではなく公式デー
  タそのものなので、他4カテゴリより信頼度が高い。
- **他4カテゴリ(安全保障・感染症・インフラ・治安)はキーワード頻度**: ORACLE本体と同じ「記事件数
  を飽和カーブでスコア化」する手法。ニュース報道量の代理指標であって、検証済みの実測値ではない
  ことをUI上の免責文で明示している。
- **重み**: `Disaster 0.35 / Security 0.25 / Health 0.15 / Infrastructure 0.15 / PublicSafety 0.10`。
  「暮らしへの直接的な脅威」という趣旨上、災害の重みを意図的に最大にしている。ORACLE本体(世界の
  地政学的緊張)とは重みの思想自体が異なる。
- **JMA/NHK取得ロジックはJAPAN NOWから移植**: 47都道府県分のoffice走査、特別警報コード表、Hokkaido/
  Okinawaの表示グルーピング、定時降灰予報の除外ロジックなど。JAPAN NOW側で既に本番検証済みのロジ
  ックをそのまま再利用している。

## 修正履歴

- **v0.1.1**: 本番デプロイ直後、`Security`が8記事未満で即カンスト(100)する事象を確認。原因は
  `軍事`・素の`自衛隊`という汎用語が、防衛予算や訓練など有事性のない通常報道まで拾っていたため。
  実際に事件・侵犯・発射など「起きたこと」を示す語だけに絞った。同じ理由で`Health`の素の`流行`
  (ファッション等の「流行」とも一致してしまう)も`感染拡大`に置き換え。Google News検索クエリ
  (`GOOGLE_NEWS_JP_QUERY`)も同じキーワードセットに揃えてある。

- **v0.1.2**: 実運用で「注意報しか出ていない平常な夏日なのにCRITICAL(74)」という事象を確認。原因は
  複合的だった:
  1. `感染症`が定例のサーベイランス報告・市場調査レポート・ワクチン研究拠点のニュースまで拾って
     いた → `scoreHealthCategory`を新設し、「病名(インフルエンザ等)」と「悪化を示す語(急増・拡大・
     クラスター等)」が同じ見出しに両方ないとカウントしない方式に変更。
  2. 北朝鮮DMZの地雷爆発(韓国軍発表)が`爆発`という汎用語で「治安・事件事故」に紛れ込んでいた →
     JAPAN NOWの「海外限定ストーリー除外」フィルタ(`isForeignOnlyStory`)を移植し、Health/
     Infrastructure/PublicSafetyにのみ適用(Securityは北朝鮮・中国の動向を意図的に含むため対象外)。
  3. 1つの実イベント(根室市の断水)を4媒体が別の見出しで報じただけで、そのカテゴリが即100点に
     達していた → `newsCategoryScore`の飽和分母を8→14に緩和。根本的な「同一イベント検出」ではな
     く緩和策であることをここに明記しておく。
  4. 最大の原因: 注意報(最も軽い警報レベル)は夏の午後、日本のどこかで毎日のように出ており、重み
     2点/件のままだと平常日でも25件×2点=50点(WATCH〜HIGH相当)に達していた → 重みを0.4点/件に
     引き下げ、実際の警報・特別警報が出たときだけスコアが跳ねるように調整。
  同じ状況を新ロジックで再計算すると、総合指数は74(CRITICAL)→約25(STABLE寄り)に低下する。

## 未検証・今後の課題

- このサンドボックスはネットワークアクセスがないため、JMA/NHK/Google Newsへの実際の疎通確認は
  **未実施**。デプロイ後、まず`dataStatus`が`LIVE`になるか、`sourceError`に何も出ていないかを確認
  すること。
- キーワードリスト(`CATEGORY_KEYWORDS`)は初期案。実際の報道でどの程度誤検知/検知漏れがあるか、
  本番投入後にヘッドラインを見ながら調整が必要。
- **地図(v0.1.5)は未実機確認**: unpkg経由でのjson取得自体は`web_fetch`ツールで内容を確認済みだが、
  ブラウザから実際に`https://unpkg.com/jpn-atlas@1.0.2/japan/japan.json`を`fetch()`した際のCORSヘッダ
  ーや読み込み速度、`d3.geoPath()`での実際のレンダリング結果はこの環境で検証できていない。デプロイ
  後に地図が実際に日本の形で表示されるか、都道府県の色分けが警報データと正しく対応しているか確認
  すること。unpkgが不安定な場合はjsdelivr等の別CDN経由での配信に切り替える必要があるかもしれない。
- `newsCategoryScore`の分母緩和(8→14)は「同一イベントの多媒体重複報道」問題への根本対応ではなく
  緩和策。本当の解決には見出しからの固有表現抽出(地名+事象など)によるイベント単位のグルーピング
  が要るが、今回は実装していない。単一の大規模実イベントが再び高スコアを牽引したら、この点を再検討
  すること。
- `isForeignOnlyStory`の`JAPAN_RELEVANCE_TERMS`に`県`/`市`を含めているため、日本の都道府県・市名
  が入っていればほぼ確実に「国内」と判定される。裏を返すと、日本のどこかの県名を偶然含む海外ニュ
  ース(稀だが)は除外されない可能性がある。
- Redis永続化(24Hデルタ)はORACLE本体と同じ環境変数(`KV_REST_API_URL`/`KV_REST_API_TOKEN`など)を
  流用する設計。ORACLE本体と同じUpstash DBを共用する場合はキー名(`oraclejapan:score_log`)が衝突
  しないことを確認済み。

## デプロイ

Vercelにこのフォルダをそのままpushすればよい(ORACLE本体・JAPAN NOWと同じ構成)。
Redis永続化を使う場合はVercel Marketplace経由でUpstash Redis統合を追加し、環境変数を注入する。

## テスト

```
npm test
```

ネットワーク呼び出しを伴わない純粋関数(スコア計算・重複排除・警報コード解釈など)のみをテスト
している。実際のfetch/パース部分はデプロイ後に本番データで確認する必要がある。
